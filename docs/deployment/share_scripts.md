# Sharing Your Scripts

Once a [build script](../advanced_usage/custom_app.md), [data script](../advanced_usage/custom_data.md), or [helper script](../advanced_usage/custom_helper.md) works, it's worth sharing — a script in a shared source is one build that serves everyone, instead of each person rediscovering the same vendor tarball.

Three ways to do it:

| Option | Reaches | Setup |
|---|---|---|
| [Open a PR](#option-1-open-a-pr-against-cnt-scripts) | everyone using CondaTainer | none |
| [Shared directory](#option-2-a-shared-directory) | the same filesystem | up to one env var |
| [Your own URL](#option-3-your-own-url) | anywhere over HTTP | serve the layout, generate metadata |

- Publish generally useful tools and public data upstream.
- For everything internal, a shared directory is the simplest: no server, edits are live immediately.
- Reach for a URL only when your users are spread across machines that don't share storage.

CondaTainer merges every configured source, so you can keep internal scripts in a shared directory *and* upstream the general ones.

```{tip}
Before sharing either way, drop the script into a local `build-scripts/` directory and build it once. See [Where Build Scripts Live](../advanced_usage/custom_app.md#where-build-scripts-live).
```

## Option 1: Open a PR Against cnt-scripts

[cnt-scripts](https://github.com/Justype/cnt-scripts) is the default source every CondaTainer install reads, so a script shows up in everyone's `condatainer avail` with no configuration.

1. Fork [Justype/cnt-scripts](https://github.com/Justype/cnt-scripts) and branch off `main`.
2. Add your file at the path that matches its [naming convention](../manuals/build_script.md#naming-conventions).
3. Build it and confirm it works.
4. Open a PR against `main`, including the terminal output or job log from your successful build.

See [CONTRIBUTING.md](https://github.com/Justype/cnt-scripts/blob/main/CONTRIBUTING.md) for the full checklist.

```{note}
Metadata is generated by CI on merge — don't hand-edit `metadata/`.
```

## Option 2: A Shared Directory

For scripts that stay inside one cluster, a directory on shared storage beats any remote — CondaTainer scans local script directories directly, so there's **no metadata to generate and nothing to publish**. Save a file and it's available to everyone immediately.

The directory layout is always the same:

```
<cnt_root>/
├── build-scripts/          # app / data / OS scripts
│   └── my-tool/1.0
├── helper-scripts/         # helper scripts
│   └── my-service
└── images/                 # built .sqf overlays (optional, see below)
```

When CondaTainer itself lives in the shared directory, there's **nothing to configure — just copy the file in.** It detects its own install root from the binary's location: with the binary at `/shared/labA/condatainer/bin/condatainer`, the root is `/shared/labA/condatainer`, and its `build-scripts/`, `helper-scripts/`, and `images/` are already in the search path for everyone using it. See [Shared Group Installation](./group_install.md) for setting this up.

```bash
mkdir -p /shared/labA/condatainer/build-scripts/my-tool
cp my-tool/1.0 /shared/labA/condatainer/build-scripts/my-tool/1.0
condatainer avail my-tool     # already visible
```

If CondaTainer is installed elsewhere — a system-wide install, or a binary in a personal `bin/` — point it at the directory with `$CNT_EXTRA_ROOT`; see [Group Use](./system_install.md#group-use-under-system-install) and [Configure It Once for the Whole Group](#configure-it-once-for-the-whole-group).

```{tip}
Keep the shared directory in a git repo checked out on the filesystem. You get review and history, and a `git pull` publishes to everyone at once.
```

## Option 3: Your Own URL

Use a remote source when your users are on machines that don't share a filesystem — several clusters, or a lab spread across institutions.

**It does not have to be GitHub.** CondaTainer just fetches fixed paths under a base URL, so any plain HTTP(S) server works: an internal web server, an object-store bucket, a GitLab or Gitea raw endpoint, static hosting. All it needs is to serve the layout below and be reachable from your users' machines.

### 1. Mirror the cnt-scripts layout

```
<base URL>/
├── build-scripts/          # app / data / OS scripts
│   ├── cellranger/9.0.1
│   └── grch38/genome/gencode
├── helpers/                # helper scripts
│   └── my-service
└── metadata/               # generated — see below
    ├── build-scripts.json.gz
    └── helper-scripts.json.gz
```

### 2. Generate the metadata

`condatainer avail` and `condatainer helper --list` read the **gzipped metadata files**, not the scripts themselves. A source without them is invisible.

Copy the generators from cnt-scripts and run them whenever a script changes:

- [`generate_metadata.py`](https://github.com/Justype/cnt-scripts/blob/main/.github/scripts/generate_metadata.py) — writes `metadata/build-scripts.json.gz`
- [`generate_helper_metadata.py`](https://github.com/Justype/cnt-scripts/blob/main/.github/scripts/generate_helper_metadata.py) — writes `metadata/helper-scripts.json.gz`

If your host is a git forge, automate it — cnt-scripts' [workflows](https://github.com/Justype/cnt-scripts/tree/main/.github/workflows) regenerate and commit on every push. Otherwise run them as part of whatever publishes the files. Forgetting this step is the usual reason a new script never appears.

### 3. Point CondaTainer at it

The base URL is whatever prefix makes `<base>/metadata/build-scripts.json.gz` resolve — for GitHub that's the raw content root:

```bash
# Any HTTP(S) server that serves the layout
condatainer config prepend extra_scripts_links https://scripts.labA.edu/cnt

# GitHub raw content root
condatainer config prepend extra_scripts_links \
    https://raw.githubusercontent.com/MyOrg/my-scripts/main
```

Then refresh and confirm your scripts appear:

```bash
condatainer update       # refresh build + helper metadata from every source
condatainer avail my-tool
condatainer helper --list
```

Sources are searched in order — `extra_scripts_links` entries first (earliest wins), then `scripts_link`. So a script in your source **shadows** a same-named one in cnt-scripts, which is how you override an upstream script.

```{tip}
`scripts_link` *replaces* the default source rather than adding to it. Set it only if you want to cut cnt-scripts out entirely; otherwise use `extra_scripts_links`.
```

```{note}
CondaTainer fetches the URL unauthenticated, so anything behind a login or token won't work — a private GitHub repo included. Either serve the files somewhere your users can read without credentials, or use a [shared directory](#option-2-a-shared-directory) instead.
```

## Configure It Once for the Whole Group

Skip this if CondaTainer lives in the shared directory — that setup needs no configuration at all.

Otherwise, don't ask each person to run config commands. Put the settings in a `config.yaml` at the shared root and have users set `$CNT_EXTRA_ROOT` to it (one line in a module file or a shell profile), or ship the whole thing as an environment module:

```yaml
# /shared/labA/condatainer/config.yaml
extra_scripts_links:
  - https://scripts.labA.edu/cnt
build:
  ncpus: 8
```

Scaffold it with:

```bash
CNT_EXTRA_ROOT=/shared/labA/condatainer condatainer config init -l extra-root
```

`$CNT_EXTRA_ROOT` already contributes the shared `build-scripts/`, `helper-scripts/`, and `images/` directories, so this file only needs settings that aren't implied by the layout — remote sources, build defaults, and so on.

Array keys merge across config layers, so this adds your sources without overriding anything a user set themselves. See [Multi-Tier Setup](../manuals/configuration.md#multi-tier-setup-system--group--user).

## Sharing Built Overlays

A script still costs everyone a build. If you have shared storage, share the `.sqf` too — it's a single file, and CondaTainer finds it in any configured images directory:

```yaml
extra_image_dirs:
  - /shared/labA/condatainer/images
```

`condatainer list` then shows the overlay as installed for everyone, with no rebuild. This is worth doing for anything expensive: large data overlays, indexes, apps with long builds.
